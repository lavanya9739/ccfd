About the Project

AI-powered credit card fraud detection using LightGBM, built with Flask for fast and reliable deployment.

History of LightGBM

Origins

LightGBM (Light Gradient Boosting Machine) was developed by researchers at Microsoft and Peking University to overcome the efficiency and scalability challenges of XGBoost when handling high-dimensional input features and massive datasets.

Key Innovation: GOSS

Gradient-Based One-Side Sampling (GOSS) focuses on instances with larger gradients, ensuring faster and more accurate learning while retaining data with large gradients and randomly sampling from data with small gradients.

Key Innovation: EFB

Exclusive Feature Bundling (EFB) groups mutually exclusive features to reduce dimensionality without losing information, dramatically reducing computation time.

Histogram-Based Approach

Uses discrete bin values to define the best split points instead of sorting all continuous feature values, providing high speed and lower computational costs compared to other algorithms.

Leaf-Wise Growth

Unlike level-wise growth in XGBoost, LightGBM grows trees leaf-wise, choosing the leaf with the maximum delta loss to grow, resulting in lower loss and higher accuracy.

Why LightGBM Stands Out

Speed & Efficiency

Histogram-based learning where continuous values are bucketed into discrete bins, making it faster than traditional gradient boosting algorithms.

Memory Efficient

Highly memory-efficient due to its histogram-based approach and data compression techniques, more efficient than XGBoost.

Large Datasets

Scales well with large datasets in both training speed and memory usage, making it ideal for large-scale financial problems.

Categorical Features

Handles categorical features natively, saving computation time and memory by eliminating the need for one-hot encoding.

Distributed Learning

Supports parallel and distributed learning, efficiently handling massive datasets by splitting computations across multiple machines.

Data Preprocessing & Feature Engineering

1

Categorical Encoding

Transaction Type, Currency Code, Transaction Country, and Transaction City are encoded into numerical values using Label Encoding. For example: Online Payment becomes 1, INR becomes 0, IN becomes 0, Chennai becomes 0.

2

Numerical Features

Transaction Amount, Credit Limit, Merchant Category Code, and Open to Buy are continuous numerical values used directly as features in the model.

3

Feature Scaling

Standard Scaler is applied to normalize all features to have zero mean and unit variance, ensuring that no feature dominates due to its scale.

4

Feature Alignment

Columns are aligned with the training data to ensure that the model receives features in the exact order and format it was trained on.

Model Training with LightGBM

Prediction Formula

y = Sum(m=1 to M) of alpha_m * T_m(x)
  • y = predicted label (fraud or not fraud)
  • M = total number of trees
  • alpha_m = weight (importance) of each tree
  • T_m(x) = the m-th decision tree making predictions based on input features

Loss Function (Binary Cross-Entropy)

L(y, p) = -[y * log(p) + (1 - y) * log(1 - p)]
  • L = loss (error)
  • y = true label (0 for not fraud, 1 for fraud)
  • p = predicted probability (between 0 and 1)

Prediction & Decision Threshold

After training, the model predicts a probability score between 0 and 1. The decision threshold determines the final classification:

Not Fraud
Fraud
0.5
Low Risk (0 - 0.3)
Medium Risk (0.3 - 0.5)
High Risk (0.5 - 1.0)

Example

If the model predicts a probability of 0.72 for fraud, since 0.72 > 0.5, the transaction is classified as Fraud.
